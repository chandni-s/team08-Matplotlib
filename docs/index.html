Matplotlib

Matplotlib is a program created by John D Hunter to plot graphs and other functions similar to MATLAB. The program is an open source on github and has been going strong even though John died in 2012.

Matplotlib is broken up into three main layers which builds on top of each other. These layers are, from top to bottom respectively: the Scripting Layer, the Artist Layer, and the Backend Layer. Each one of these layers is abstracted from each other and does not rely on the any of the layers above it.

The scripting layer: is the interface between the user and the matplotlib library, mainly using the pyplot module. Pyplot provides interaction with the classes in the layers below it. This allows the user to easily generate and manipulate graphs through simple method calls, such as plotting a graph like below: 

import matplotlib, pyplot as plt
                            
plt.plot();

PyPlot is an implementation of a stateful program which helps with its memory usage. It does so by using different states to keep track of things like current figure, canvas area and current axes.

Transition Scripting Layer -> Artist Layer: PyPlot imports all classes of the artist layer such as Axes, subplots, Lines, Artist, Figure and others which it uses in its functions to tell the machine what to draw. It does so through appropriate sequences of method calls to the respective classes. This translate to a high degree of coupling from the Scripting Layer to the Artist Layer.

The Artist Layer: The main goal of the artist layer is to draw the graph based on parameters passed in the scripting layer. It consists of 2 different types of artists, primitives and containers. Primitive objects represents the basic images needed to make graphs such as Line 2D, Rectangle, Text and other images. Container objects represents where you draw the primitives on, such as Axis, Axes, and Figure.

In the Artist layer, the Figure class is the base class needed to instantiate everything else; it is the window on which Axes and subplots are drawn. The Axes class contains most of the elements such as Text, Line, Axis and Ticks. Each Axes object has a subplot object assigned to it, which is used to manipulate properties of these elements. The subplot class is what actually controls the graph on the screen and as people change parameters and elements of a graph, they call functions within subplot and Axes class.

Transition Artist Layer -> Backend Layer: When the artist wants to render the graph on the screen, it calls on the Backends_bases module which is the gateway to the backends package that renders the pictures. The coupling between these two layers are low.

The Backend layer: is the layer which handles rendering the actual graph on screen. The backend package handles displaying all the physical properties of lines, such as colors and plots to show up on the screen. The classes backend_bases.FigureCanvasBase, backend_bases.RendererBase and backend_bases.GraphicsContextBase implement the backend layer to handle creating the canvas, rendering the images, and rendering the colors respectively.

Potential Improvements

1) We've noticed that there were numerous imports from the Scripting Layer (pyplot) to the Artist Layer (Axes, Labels, etc). This could cause problems if any of the classes depended on gets deprecated or refactored. Also, if there was a desire to implement a new interface in place of pyplot, it could be an ordeal to ensure every needed dependencies are imported. An interface in the Artist Layer could reduce the level of coupling between it and the Scripting Layer.

2) In the factory design pattern involving _Subplots.py to generate AxesSubplots objects, Subplot_class_factory generates different types depending on input; however, the generated classes is always inherting from SubplotBase. If new parent objects were introduced, the current design would require additional condition statements, which does not follow the Open/closed principle of Object-Oriented methodology.

Design Patterns that we found

The Scripting layer first sends to the backend to set up the renderer by updating and resizing based on the the figure. The FigureCanvas classes are backend representations of the window. Think of it as the Paper the artist draws on. From there it is sent to the allow rasterization function in the Artist file which begins the drawing process

Decorator

When it comes time to draw the the actual figure a Decorator pattern is used. The Decorator is important to allow customization to your chart. With a Decorator you can have a chart with any permutation of lines, images, text, legends, etc. Each part of the figure has access to the Artist class located here (lines were only drawn connecting figure becasue when creating a new window the Artist begins with the Figure class, but all the classes on the right extend Artist, when it needs to be drawn it hands its draw() method to Artist which in turn gives it to the renderer to actually draw. Think of the Renderer as the paintbrush and the Artist as the one who knows how to paint. This Decorator spans two layers: the Artist and the Backend. The order that objects are drawn is determined by a value in each of the classes called zorder, and the actual drawing takes place in a function call allow_rasterization located in artist.py.

There is actually a lot of setup that must go into the backend before the actual drawing can occur. To help your understanding you can look at the sequence diagram (the second diagram) to the right that shows and explains the execution of a plt.show() command.

Factory

This Factory is used when adding subplots to a figure. A figure can contain multiple Axes which contain subplot information defined in the subplotBase class. When the user adds a new subplot, the current figure's add_subplot method will run which in turn asks for a new instance of a subplot. The SubplotBase class is meant to be used as an interface. We don't want the current Figure's Axes to contain an instance of SubplotBase. So instead matplotlib dynamically creates an instance of the class AxesSubplot which will have all the methods of SubplotBase. Both the SubplotsBase class and the Factory Method(subplot_class_factory) can be found here

Note that the _Subplots class diagram does not represent a class. _Subplots is a file that holds the Subplot_class_factory() function. For the sake of simplicity and neatness we made it a class diagram.

Adapter

Not all users of matplotlib know how to use NumPy arrays and lists, but matplotlib depends heavily on NumPy. So to answer this, matplotlib allows users to give regular python lists as input to all their plotting functions. The example in the side image uses the _process_plot_var_args class located here which contains methods that puts the data to be plotted in a format that the Artist can recognize. These methods call the methods of CBook which is a set of files that will be our Adapter. The methods of CBook that are called can take Python lists or NumPy arrays depending on what the user inputed, and in either case a NumPy array is returned.
